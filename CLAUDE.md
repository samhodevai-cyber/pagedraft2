# CLAUDE.md — 제휴 콘도 예약 페이지 프로젝트

> 이 파일은 작업 컨텍스트를 보존하기 위한 문서다. 새 세션에서도 이 파일만 읽으면
> 지금까지의 대화/결정/구현 상태를 그대로 이어서 작업할 수 있도록 정리해 둔다.

## 0. 프로젝트 개요

- **목적**: 회사와 제휴된 콘도를 직원이 웹에서 조회·예약 신청하고, 총무부 관리자가
  신청을 승인/반려 처리하는 사내 페이지.
- **현재 단계**: 백엔드 없이 동작하는 **단일 HTML 인터랙티브 프로토타입(목업)**.
  모든 로직은 순수 HTML/CSS/Vanilla JS 한 파일에 들어 있고, 데이터는 JS 내 mock
  `api` 레이어로 관리한다. (추후 React + REST API/DB/인증/메일로 확장 예정)
- **작업 디렉터리**: `c:\dev\page` (git 저장소 아님, Windows 11 / PowerShell 환경)
- **요구사항 원본**: [prd.md](prd.md) — 회사가 준 구축 요청서.

### 초안 저장 규칙 (사용자 지정, 반드시 지킬 것)
- 초안은 항상 `origin\` 폴더에 저장한다.
- **사용자가 새로운 작업(기능 추가/수정)을 요청할 때마다 매번 새 버전의 html 파일을 만든다.**
  기존 파일을 직접 수정하지 않고, 그 파일을 복사해 새 파일로 만든 뒤 그 위에 변경사항을 반영한다.
- **같은 파일명이 이미 있으면 파일명 뒤에 넘버링**해서 구분한다.
  예) `index.html` 존재 → 다음 초안은 `index2.html`, 그 다음은 `index3.html`, `index4.html` …
- 기존 초안은 절대 덮어쓰지 않고 보존한다(버전 히스토리 용도). 가장 번호가 큰 파일이 항상 최신.

### 배포 규칙 (턴13에서 확정, 반드시 지킬 것)
- 이 프로젝트는 GitHub 저장소 [samhodevai-cyber/pagedraft2](https://github.com/samhodevai-cyber/pagedraft2)에
  연결되어 있고, GitHub Pages로 배포되어 있다.
  **배포 URL: https://samhodevai-cyber.github.io/pagedraft2/**
- 저장소 루트의 `index.html`이 GitHub Pages가 서빙하는 진입 파일이다(`origin/`이 아님).
- **새로운 작업을 완료해 새 버전 파일(`origin/indexN.html`)을 만들 때마다, 사용자가
  "매번 자동 반영"을 선택했으므로 다음을 반드시 같이 수행한다:**
  1. 그 최신 파일을 루트 `index.html`로 복사(덮어쓰기)한다.
  2. `git add -A && git commit -m "..." && git push`로 GitHub에 반영한다.
  3. 배포 URL이 실제로 갱신됐는지 `curl`로 상태코드/타이틀 정도는 확인한다.
- git 커밋 identity는 이 저장소에 로컬로 설정되어 있다
  (`user.name=samhodevai-cyber`, `user.email=samhodevai@gmail.com`).
- **인증/토큰 관련 주의**: `git push`는 Git Credential Manager(번들, `credential.helper=manager`)로
  이미 로그인되어 있어 추가 입력 없이 동작한다. 단, `git credential fill` 등으로 얻은 실제 토큰 값을
  curl 명령줄 인자(`-H "Authorization: Bearer $TOKEN"` 등)에 직접 넣는 행위는 자동 모드 보안
  분류기가 "자격 증명 노출"로 차단한다(턴13에서 실제 발생) — 시도하지 말 것. GitHub Pages
  설정처럼 API 호출이 필요한 관리 작업은 대신 사용자에게 웹 UI에서 직접 클릭하도록 안내한다.
- **데이터 공유 한계는 배포 후에도 동일**: GitHub Pages는 정적 호스팅이라 `localStorage`가
  방문자 브라우저별로 분리된다. 여러 사람이 실제로 신청→승인 흐름을 같이 쓰려면 별도
  서버+공유 DB 연동(prd.md 7장)이 필요함을 매번 배포 안내 시 상기시킬 것.

## 6. 코드 구조 규칙 / 컨벤션 (이어서 작업할 때 지킬 것)

- **단일 HTML 파일**: `<style>` + 마크업 + `<script>` 한 파일. 외부 의존성/빌드 없음.
- **mock `api` 객체**가 모든 데이터 CRUD를 담당. 화면 코드는 `api`만 호출
  (직접 배열 조작 금지) → 백엔드 교체 지점이 명확하도록.
- 날짜는 `"YYYY-MM-DD"` 문자열. `dateRange/nights/ymd/pad` 유틸 사용.
- 데모 기준 날짜는 **2026-06-30(오늘)**, 달력 초기 표시는 **2026년 7월**로 하드코딩.
  (오늘 날짜 = `currentDate` 컨텍스트 기준. 실제 백엔드 연동 시 동적으로 교체)
- 상태 한글 라벨: 접수 신청중 / 신청 확인중 / 승인 / 반려. 뱃지 클래스 매핑 `badge()` 함수.
- UI 텍스트는 한국어. 토스트 = 가벼운 알림, info modal = 명시적 안내 팝업,
  reject modal = 반려 사유 입력, confirm modal(`confirm-overlay`/`confirmPopup()`) =
  삭제 등 되돌리기 어려운 동작의 확인/취소(턴8 신규, 재사용 가능한 공용 컴포넌트).
- CSS 변수(:root)로 색상/라운드/그림자 토큰화. 브랜드색 `--brand:#3b5bdb`.
- **데이터 변경 메서드는 반드시 끝에 `persist()` 호출**(턴10). 새 mutating 메서드를
  추가할 때 빠뜨리지 말 것 — 안 넣으면 그 변경만 새로고침/다른 탭에서 사라짐.
- **`serve.ps1`처럼 한글 주석/문자열이 포함된 파일은 절대 Bash의
  `powershell.exe -Command "(Get-Content ...) -replace ... | Set-Content ..."`로
  고치지 말 것.** 인코딩이 깨져 한글이 모지바케(mojibake)된다(턴12에서 실제 발생,
  `Write` 도구로 복구함). 항상 Edit/Write 도구를 사용할 것 — Bash/PowerShell의
  `Set-Content`는 이 환경에서 안전한 UTF-8 인코딩을 보장하지 않는다.

## 7. 아직 mock / 미구현 (prd.md 7장 — 향후 작업 후보)

1. React(또는 선호 스택) 기반으로 재구성.
2. 실제 백엔드 REST API + DB 연동(인증, 예약 CRUD, 마감일 관리).
3. 사번+성명+비밀번호 → 추후 사내 **SSO** 확장 가능 구조.
4. **날짜 중복 방지**: 프론트 체크 + 백엔드 승인 엔드포인트 재검증(현재 mock으로만 시뮬레이션).
5. 이메일 실제 발송(SMTP/SendGrid/AWS SES) — 현재 `sendEmail()` mock.
6. 비밀번호 해시(`password_hash`), 세션/토큰 등 보안 처리.
7. **데이터 영속성 — index7.html부터 localStorage로 부분 개선됨(턴9→턴10)**:
   index6.html까지는 `api` IIFE 메모리에만 데이터가 있어 새로고침/다른 탭에서
   초기화됐음. index7.html부터는 모든 변경 메서드가 `persist()`로 `localStorage`
   (`condo_app_mock_data` 키, `STORAGE_VERSION`로 스키마 버전 체크)에 자동 저장하고,
   시작 시 그 값을 우선 불러온다. **단, localStorage는 origin 단위 공유**라서:
   - `serve.ps1` 로컬 서버(같은 host:port)로 열면 index7 이후 버전 파일 간에도
     데이터가 공유됨.
   - **`file://`로 직접 열면 브라우저가 파일별로 다른 origin으로 취급할 수 있어
     버전 파일 간 공유가 보장되지 않음** — 이 경우 새로고침까지는 유지되지만
     다음 버전 파일을 열면 다시 시드 데이터로 시작할 수 있음.
   - 데이터 구조(필드)를 바꾸는 작업을 하면 `STORAGE_VERSION`을 올려서 구버전
     저장값이 새 코드와 안 맞는 상태로 섞이지 않게 할 것.
   - 관리자 '콘도 관리'에 "데모 데이터로 초기화" 버튼(`api.resetDemoData()`)으로
     언제든 시드 데이터로 되돌릴 수 있음.
   - 진짜 영속성(여러 사용자/기기 공유)은 여전히 백엔드 연동(2번)이 필요함.

## 8. 사용자 작업 선호 (관찰된 패턴)

- 요청은 한국어. 결과 HTML은 브라우저로 바로 열어 확인.
- **새로운 작업을 요청할 때마다 매번 새 html 파일을 버전 넘버링으로 만들 것**
  (기존 파일을 직접 수정하지 않음 — 턴7에서 사용자가 명시적으로 재확인).
  같은 파일명이 있으면 파일명 뒤에 숫자를 붙여 구분.
- 요구사항을 "[공통]/[직원]/[관리자]" 형태로 구획해서 전달하는 경향.
- 기능 추가 요청 시 기존 동작은 유지한 채 델타만 반영.
- 버그 리포트를 받으면 추측으로 고치지 말고 로컬 서버(`serve.ps1`) +
  Claude Preview 도구로 **직접 재현 후** 원인을 특정하고 고치는 패턴을 선호함(턴5).
- 이메일 수신처 등 사용자가 불러준 값에 오타로 의심되는 부분이 있으면, 일단 그대로
  반영하되 확인을 요청한다(임의로 고치지 않음 — 턴6의 `dhhwang@samhodev.co.rk` 사례).
- 워크플로우 자체에 대한 개선 제안("이거 번거로운데 방법 없을까?" 같은 탐색형 질문)에는
  먼저 2~3문장으로 권장안 + 트레이드오프만 제시하고, 사용자가 동의("응" 등)한 뒤에
  구현한다(턴10) — 묻지마 구현 금지.
