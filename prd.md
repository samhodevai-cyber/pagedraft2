# 제휴 콘도 예약 페이지 — 구축 요청서

## 1. 개요
회사와 제휴된 콘도를 직원들이 웹에서 조회하고 예약 신청할 수 있는 사내 페이지를 만든다.
직원 화면과 총무부 관리자 화면, 두 가지로 구성한다.

## 2. 직원 화면

### 2.0 최초 접속 / 로그인
- 페이지 최초 접속 시 콘도 목록보다 먼저 로그인 화면이 표시된다.
- 입력 항목: 사번, 성명, 비밀번호
- "확인" 버튼을 누르면 서버에서 사번+비밀번호를 검증하고, 일치하면 콘도 목록 화면으로 이동한다.
- 입력값이 비어있거나 인증에 실패하면 에러 메시지를 표시한다.
- 로그인 성공 시 이후 화면(예약 신청 등)에 성명/부서/사번 정보를 함께 사용한다.
- (참고) 장기적으로는 사내 SSO 연동으로 대체 가능하나, 1차 구축 단계에서는 사번+성명+비밀번호 기반 자체 인증으로 진행한다.

### 2.1 콘도 목록 (카드)
- 검색/필터 기능은 두지 않는다. 로그인 직후 전체 제휴 콘도를 카드 목록으로 보여준다.
- 카드에 포함되는 정보: 콘도 사진, 이름, 위치 설명. 그 외 정보(잔여 객실, 가격 등)는 카드에 노출하지 않는다.
- 카드를 클릭하면 해당 콘도의 상세 화면(2.2, 2.3)으로 이동한다. 목록 화면에서 "뒤로" 버튼으로 다시 카드 목록으로 돌아올 수 있어야 한다.

### 2.2 콘도 상세 화면 — 날짜 선택 (달력)
- 카드 클릭 시 이동하는 화면 상단에 선택한 콘도의 이름, 사진/아이콘, 위치 설명을 다시 보여준다.
- 그 아래에 월간 달력이 표시된다.
- 콘도/객실 타입을 선택하면 월간 달력이 표시된다.
- **이미 다른 직원이 예약(승인된 건)한 날짜는 빨간색으로 표시되고 클릭이 불가능하다.** 이게 핵심 요구사항이다 — 같은 날짜에 중복 예약이 발생하면 안 된다.
- 마감 날짜는 콘도(객실)별로 독립적으로 관리한다. 한 콘도의 마감 날짜가 다른 콘도의 예약 가능 여부에 영향을 주지 않는다.
- 체크인 날짜를 먼저 클릭하고, 체크아웃 날짜를 클릭하면 두 날짜가 선택된다.
- 선택 구간 안에 마감된 날짜가 포함되어 있으면 선택을 초기화한다.
- 범례: 예약 가능(기본), 마감(빨강), 선택중(강조색).

### 2.3 예약 신청 폼
- 체크인/체크아웃 날짜(달력에서 선택된 값 자동 반영)
- 객실 타입 선택
- 인원수 입력
- 신청 사유 / 비고 (선택 입력)
- 연 1회 이용 제한 등 사내 정책이 있다면 안내 문구 표시
- 제출 시 상태는 "접수중"으로 시작한다.

### 2.4 내 예약 신청 현황
- 본인이 신청한 예약 목록을 상태와 함께 보여준다.
- 상태: 접수중 / 확인중 / 승인 / 반려
- 반려된 건은 반려 사유를 함께 표시한다.
- 접수중/확인중 상태의 예약은 직원이 직접 취소할 수 있다.

## 3. 총무부 관리자 화면

### 3.1 접근 권한
- 총무부 관리자 권한을 가진 계정만 접근 가능해야 한다 (역할 기반 접근 제어 필요).

### 3.2 신청 현황 대시보드
- 상태별 건수를 요약 카드로 보여준다: 접수중 / 확인중 / 승인 / 반려

### 3.3 신청 목록 및 처리
- 모든 직원의 예약 신청을 목록으로 보여준다 (신청자, 부서, 이메일, 콘도, 날짜, 인원, 상태).
- 각 신청 건에 대해 다음 액션을 제공한다:
  - **확인중으로 변경**: 검토 중임을 표시
  - **승인**: 상태를 승인으로 변경하고, 해당 날짜를 자동으로 마감(다른 직원이 선택 불가) 처리. 승인 시 신청자 이메일로 승인 안내 메일을 발송한다.
  - **반려**: 반려 사유를 입력하는 모달을 띄우고, 사유 입력 후 확정하면 상태가 반려로 바뀌고 사유가 함께 저장된다.

### 3.4 이메일 발송
- 승인 처리 시 신청자 이메일로 자동 발송한다.
- 메일 내용: 콘도명, 체크인/체크아웃 날짜, 인원, 승인 처리자 등 포함.
- 브라우저에서 직접 발송할 수 없으므로 서버 사이드에서 SMTP 또는 이메일 발송 API(SendGrid, AWS SES, 사내 메일서버 등)를 호출해야 한다.

## 4. 핵심 비기능 요구사항

- **날짜 중복 방지(동시성)**: 두 명이 거의 동시에 같은 날짜를 신청할 경우를 대비해, 예약 확정(승인) 처리 시 서버/DB 레벨에서 해당 날짜가 이미 마감되지 않았는지 다시 한번 확인 후 처리해야 한다. 단순 프론트엔드 체크만으로는 race condition을 막을 수 없다.
- **권한 분리**: 일반 직원과 총무부 관리자의 화면/기능이 분리되어야 한다.
- **인증**: 사번 + 성명 + 비밀번호 입력 기반 로그인으로 시작하며, 추후 사내 SSO 연동으로 확장 가능한 구조로 설계한다.
- **반응형 레이아웃**: PC와 모바일 모두에서 자연스럽게 보이도록, 화면 해상도에 따라 레이아웃이 유동적으로 바뀌어야 한다. 콘도 카드 목록은 그리드 기반으로 구현하고(`auto-fit`/`minmax` 방식 등), 화면 폭이 좁아지면 자동으로 1~2열로 줄어들도록 한다. 로그인 폼, 달력, 예약 폼 등 모든 화면 요소가 모바일 폭에서도 잘리거나 깨지지 않아야 한다.

## 6. 화면 동작 참고용 프로토타입 (HTML/JS)

아래 코드는 클로드가 시각화 도구로 만든 인터랙티브 목업이다. 실제 디자인 톤, 화면 구성, 인터랙션 흐름(달력 날짜 마감 처리, 승인/반려 플로우, 이메일 발송 토스트 등)의 참고용으로 사용하고, 실제 구현 시에는 백엔드 연동(API, DB, 인증, 메일 발송)을 추가해 완성한다.

```html
<!-- 로그인 화면: 사번 / 성명 / 비밀번호 입력 후 확인 버튼 -->
<div id="view-login">
  <input id="login-empid" placeholder="사번"/>
  <input id="login-name" placeholder="성명"/>
  <input id="login-pw" type="password" placeholder="비밀번호"/>
  <button id="login-submit">확인</button>
</div>

<!-- 콘도 카드 목록: 검색 없음, 사진/이름/위치설명만. 그리드는 auto-fit으로 반응형 처리.
     카드를 클릭하면 상세(달력) 화면으로 이동한다 -->
<div id="view-cards" style="display:none;">
  <div id="condo-cards" style="display:grid; grid-template-columns:repeat(auto-fit, minmax(220px, 1fr)); gap:14px;"></div>
</div>

<!-- 콘도 상세 화면: 선택한 콘도 정보 + 달력(마감 날짜는 빨강으로 비활성화) + 예약 신청 폼.
     "목록으로" 버튼으로 카드 목록으로 복귀 가능 -->
<div id="view-detail" style="display:none;">
  <button id="back-to-cards">목록으로</button>
  <div id="detail-header"></div>
  <div id="calendar"></div>
  <div id="booking-form">...</div>
</div>

<div style="display:flex; gap:8px; margin-bottom:1.25rem;">
  <button id="tab-employee">직원 화면</button>
  <button id="tab-admin">총무부 관리자 화면</button>
</div>

<!-- 내 예약 신청 현황: 카드 목록 또는 별도 탭에서 접근 -->
<div id="emp-list"></div>

<!-- 관리자 화면: 상태별 건수 요약, 신청 목록, 확인중/승인/반려 액션,
     반려 시 사유 입력 모달, 승인 시 이메일 발송 토스트 -->
<div id="view-admin" style="display:none;">
  <div id="status-summary"></div>
  <div id="admin-list"></div>
</div>

<div id="reject-modal" style="display:none;">
  <!-- 반려 사유 입력 -->
</div>

<div id="email-toast" style="display:none;">
  <!-- 승인 시 "OOO 으로 승인 안내 메일을 발송했습니다" 표시 -->
</div>
```

핵심 로직 요약:
- 최초 진입 시 로그인 화면을 띄우고, 사번/성명/비밀번호 검증 성공 시에만 콘도 카드 목록으로 전환한다.
- 콘도 카드 목록은 검색 없이 사진+이름+위치설명만 표시하며, `grid-template-columns: repeat(auto-fit, minmax(220px, 1fr))` 형태로 화면 폭에 따라 열 수가 자동으로 바뀐다.
- 카드를 클릭하면 해당 콘도의 상세 화면으로 전환되고(화면 전체 라우팅 또는 상태 전환), 상세 화면에는 그 콘도 전용 달력이 표시된다. "목록으로" 버튼으로 카드 목록 화면에 복귀할 수 있다.
- `bookedDatesByCondo`처럼 콘도(객실) 단위로 마감 날짜를 별도 관리하고, 달력 렌더링 시 활성 콘도의 마감 날짜만 비활성화 + 빨간색 처리한다.
- 날짜 선택은 체크인 → 체크아웃 순서의 2클릭 방식이며, 선택 구간에 마감일이 포함되면 선택을 초기화한다.
- 신청 데이터는 `requests` 배열(또는 실제로는 DB 테이블)로 관리하며 상태값을 가진다.
- 승인 액션 시 해당 콘도의 마감 날짜 목록에 구간 날짜를 모두 추가하고 이메일 발송 함수를 호출한다.
- 반려 액션 시 사유 입력 모달을 거쳐 상태와 사유를 함께 저장한다.

## 7. 클로드 코드에게 요청할 작업

1. 위 요구사항을 바탕으로 React(또는 선호 스택) 기반 프론트엔드 페이지를 구현해줘.
2. 최초 접속 시 사번/성명/비밀번호 로그인 화면을 먼저 보여주고, 인증 성공 후에만 콘도 카드 목록 화면으로 이동하도록 라우팅/상태를 구성해줘.
3. 콘도 카드 목록에는 검색 기능을 넣지 말고, 사진/이름/위치설명만 카드로 표시해줘. 카드를 클릭하면 해당 콘도의 상세(달력+예약 신청) 화면으로 이동하고, 뒤로가기로 목록에 복귀할 수 있게 해줘.
4. PC와 모바일 모두에서 자연스럽게 보이도록 반응형 그리드/레이아웃으로 구현해줘 (카드 그리드, 로그인 폼, 달력, 예약 폼 전부 모바일 폭 대응 필요).
5. 직원 화면과 관리자 화면을 라우트로 분리하고, role 기반으로 접근을 제어해줘.
6. 예약 데이터는 우선 로컬 상태/mock API로 구현하되, 추후 실제 백엔드(REST API)와 쉽게 연동할 수 있도록 데이터 페칭 레이어를 분리해줘.
7. 날짜 중복 방지 로직은 프론트엔드뿐 아니라 백엔드 API 설계 시에도 고려해줘 (예약 확정 엔드포인트에서 재검증). 마감 날짜는 콘도(객실) 단위로 독립 관리되어야 해.
8. 이메일 발송은 실제 SMTP 연동 전까지는 mock 함수로 처리하고, 추후 교체 가능하도록 인터페이스를 분리해줘.
